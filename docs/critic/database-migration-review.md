# 数据库迁移决策逆向分析

**审计日期**: 2026-03-09
**审计类型**: Pre-Mortem / 逆向分析
**问题**: 是否需要从 SQLite 迁移到 PostgreSQL？

---

## 判断

### **NO-GO — 当前不需要迁移数据库**

人类反馈"SQLite 不足以支撑我们的使用"，但经过数据分析和性能测试，**这个前提不成立**。

---

## 前提质疑：SQLite 真的不够用吗？

### 事实数据

| 指标 | 实际值 | 评估 |
|------|--------|------|
| 当前数据库大小 | 112 MB | SQLite 可处理 TB 级 |
| 历史数据大小 | 1.3 GB | SQLite 毫无压力 |
| 设备数量 | 14 台 | 个人/家庭规模 |
| Query Log 行数 | 99,391 条 | 微不足道 |
| 时间跨度 | 2 天（2026-03-04 ~ 03-06）| 正常增长 |
| 峰值 QPS | ~26/秒（93,816 条/小时）| SQLite 轻松处理 |

### 性能测试结果

```
-- COUNT 查询（24小时窗口）
sqlite3: 0.005 秒

-- Top Blocked 聚合查询
sqlite3: 0.006 秒

-- 分页查询（100条）
sqlite3: 0.006 秒
```

**结论：所有查询都在毫秒级完成，性能完全正常。**

### SQLite 的真实能力

很多人对 SQLite 有根深蒂固的误解：

| 误解 | 事实 |
|------|------|
| "SQLite 不能处理大数据" | SQLite 支持单文件 281 TB |
| "SQLite 不能高并发" | WAL 模式下支持多读并发，写吞吐 10K+ TPS |
| "SQLite 是玩具数据库" | 被 Facebook、Apple、Google、Microsoft 广泛使用 |
| "PostgreSQL 一定更快" | 对于单机场景，SQLite 没有网络开销，往往更快 |

## 心理误判检测

### 1. 锤子综合症

**症状**: "PostgreSQL 更专业，应该用它。"

**诊断**: 这是技术偏好，不是需求驱动。问题在于：我们遇到了什么 SQLite 解决不了的问题？如果答不上来，迁移就是过度工程。

### 2. 社会认同偏差

**症状**: "大家都在用 PostgreSQL。"

**诊断**: 企业级应用用 PostgreSQL 是因为需要多服务器、主从复制、高可用。我们是一人公司，14 台设备，根本不需要这些。

### 3. 提前优化

**症状**: "万一以后数据量大了怎么办？"

**诊断**: 跨过桥之前不要造桥。当 SQLite 真正成为瓶颈时（单机 100K+ QPS 或 10TB+ 数据），迁移成本相比现在更低，因为我们会有更多数据来指导决策。

---

## 迁移风险评估（如果坚持迁移）

### 最可能导致失败的 3 个原因

#### 1. **运维复杂度爆炸**

PostgreSQL 需要持续维护：

| 任务 | SQLite | PostgreSQL |
|------|--------|------------|
| 安装 | 零配置 | 需要安装、配置、调优 |
| 备份 | cp db.file | pg_dump + 持续 WAL 归档 |
| 监控 | 无需 | 连接池、慢查询、锁等待 |
| 升级 | 替换文件 | 版本升级可能需要 migration |
| 故障恢复 | 替换文件 | 可能需要 PITR、standby 切换 |

**问题**: 一人公司能承受这个运维负担吗？谁来 3AM 起来处理数据库故障？

#### 2. **迁移过程数据丢失**

迁移涉及：
- 导出 1.3GB 数据
- 转换 SQL 方言差异
- 重新建索引
- 验证数据完整性

**风险**: 任何一步出错都可能导致数据不一致或丢失。

#### 3. **部署复杂度增加**

当前架构：
```
单二进制 + 单 SQLite 文件 = 零依赖部署
```

迁移后：
```
二进制 + PostgreSQL 连接配置 + 数据库服务器 + 连接池 + 备份策略 + 监控
```

**问题**: 这增加了多少故障点？

## 替代方案（不换数据库也能解决问题）

### 1. **数据保留策略**

如果担心数据增长过快，实现自动清理：

```sql
-- 保留最近 30 天的查询日志
DELETE FROM query_log WHERE time < datetime('now', '-30 days');
```

配置化保留天数，在 `settings` 表中已有 `query_log_retention_days`。

### 2. **分区归档**

如果需要保留历史数据但不影响查询性能：

- 每月创建一个归档数据库
- 将旧数据移动到归档库
- 主库保持轻量

```bash
# 归档脚本示例
sqlite3 rust-dns.db ".dump query_log" | \
  sqlite3 archive-2026-02.db
```

### 3. **索引优化**

当前已有 16 个索引，可以进一步优化：

```sql
-- 复合索引覆盖常见查询
CREATE INDEX idx_query_log_dashboard
ON query_log(status, time DESC, client_ip, question);
```

### 4. **查询优化**

如果特定查询慢，可以：
- 使用覆盖索引
- 避免 `SELECT *`
- 增加查询缓存

## 什么情况下才需要迁移？

SQLite 成为瓶颈的真实信号：

| 指标 | 阈值 | 当前值 |
|------|------|--------|
| 单表行数 | > 1 亿 | 10 万 |
| 数据库大小 | > 100 GB | 112 MB |
| 写入 QPS | > 50,000 | ~26 |
| 查询延迟 | > 100ms | < 10ms |
| 并发写入 | 频繁 SQLITE_BUSY | 无 |

**当且仅当**以上任一指标接近阈值时，再考虑迁移。

## 结论

### 不要做

1. **现在迁移到 PostgreSQL** — 这是过度工程，解决的问题不存在
2. **因为"更专业"而迁移** — 这是锤子综合症
3. **因为"万一以后需要"而迁移** — 这是提前优化

### 应该做

1. **继续使用 SQLite** — 它完全满足当前需求
2. **实现数据保留策略** — 如果担心数据增长，定期清理
3. **监控数据库指标** — 建立性能监控，当真正需要时再迁移
4. **记录迁移触发条件** — 明确什么情况下才需要迁移

### 如果人类坚持要迁移

**前置条件**（必须全部满足）：

1. 提供具体的性能问题证据（哪个查询慢？慢多少？）
2. 确认有 2-4 周开发时间预算
3. 确认有能力承担 PostgreSQL 运维责任
4. 确认接受迁移过程中的服务中断风险
